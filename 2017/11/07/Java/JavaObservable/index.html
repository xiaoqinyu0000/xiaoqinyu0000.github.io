观察者模式

1.背景

​ “嗨！老王，听说咱们村来了一对长得很像的双胞胎？”火车上，老王跟大胖聊着八卦。

​ “哈哈，你说的是Observable和Observer吧！他们虽然长得很像，但是他们功能可大大不同哟，老大Observable可是一个标准的Java类，而老二Observer却是一个接口。

​ 千万别小看他们，一开始我还真没觉得他们有什么了不起的。直到最近，在他们的帮助下，我终于班也不用加了，觉也睡得安稳，整个人也年轻了不少！”

​ 大胖被老王说得心痒痒，思绪万千，想着如果能从老王这里讨来一两招，那自己以后就可以早点下班回去陪小美，嘴角露出了一抹笑意。

​ 老王摊开手中的本子，用笔在上面写了一个UserAction.java：

​ “村里的登录系统可是我负责的，每当用户登录或者退出的之后，我就得做好多事，就拿登录来说吧。

今天跟我说，村里的商城服务暂时不开放，让我取消调用StoreContext.init()。

明天跟我说，村里新增了社区系统，让我调用BBSContext.init(userBean)初始化社区系统。

NND！村里的一点变动都影响到我，经常半夜2-3点被叫起来协助其他服务的调试。”

2.使用Observable和Observer实现一个发布/订阅模型

“自打Observable和Observer进村之后，我就听说他们擅长处理一个类与多个类的交互。

我找到了他们。

他们了解我的苦恼之后，表示这种问题在他们那边so easy，因为他们可以帮我完成发布/订阅模型。

我表示疑虑，这种没听说过的模型能拯救我吗？

他们是这样跟我解释的：

比方说有一群人，他们想新报纸发布的时候收到通知，他们只需要去报社订阅报刊，留下自己的联系方法。

待新报发布之后，报社就按照他们留下的联系方式通知他们。

像这种多对一的关系订阅和通知就是发布/订阅模型啦

我明白了，他们的意思是，让需要用户登录或注销行为的类来我这边订阅，留下联系方式就行，一旦有用户登录，我就通过这些联系方式把消息发布出去。

我决定要试一试这种模型。

“厉害了，化主动为被动。可是，这是到底是为什么呢？”，大胖有着一种刨根问底的精神。
