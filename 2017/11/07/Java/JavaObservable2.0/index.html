观察者模式

1.背景

​ 我是一个Java类，他们都叫我UserAction。

​ 我的本职工作是负责用户登录和用户注销的操作。

​ 但随着业务迭代，我的工作重心一直不在本职工作上。

​ 一开始，用户登录之后我只需要找到数据库管理员（DBContext），把用户数据给他，他就会保存在数据库里。

​ 后来，聊天比较火，村里来了个管理聊天的小伙伴（IMContext），我得把用户数据给他初始化聊天系统：

​ 再后来，各个村都在玩电商，村长下令我们村也要搞一点电商，电商的小伙伴（StoreContext）找到我，跟我说用户登录了要把用户数据给他，他需要去获取用户订单、优惠、购物车…：

​ NND，村里搞一个活动，加一个服务，我就得跟着改动。

2.使用Observable和Observer实现一个发布/订阅模型

​ 有一次，城里的采购队来到村里，我跟他们打听到城里有一对双胞胎叫Observable和Observer，他们擅长处理一个类与多个类的交互。

​ 我决定连夜进程，夜访它们。

​ Observable和Observer听完我的叙述之后，哈哈大笑，给我讲了一个小故事：

你知道在城里，大家是怎么收到最新报纸发布的通知吗？

他们只需要去报社，把自己的号码留下来，新的报纸一来，报社就按照通讯录的号码通知大家，不管有多少人，报社才一份通讯录，报社才不管谁跟谁呢。

​ 城里的人果然先进呀，我也要搞一本通讯录，让村里需要得到用户登录消息的类来我这边留下号码，我按号码通知就行。

​ Observable满意地看着我说：“只要每月5J币，我们兄弟帮你处理所有记录号码和通知的细节”。

​ 本着时间就是金钱的原则，我跟村长申请了5J币经费。

​ 他们让我继承Observable，并在用户登录之后调用notifyObservers(Object o)方法进行通知：

​ 我们需要找一名试验者，DBContext自告奋勇。

​ 首先，它须要实现Observer接口，重写update(Observable observable, Object o)方法。

​ o参数就是就是我之前通知时的userBean。

​ 接下来，它就要来我这里登记号码了，登记号码用的是Observable提供的addObserver(Observer o)方法：

​ 我们试了N遍，每次我一调用notifyObservers(userBean)，DBContext的update方法就能接收到userBean，它们双胞胎兄弟活不赖嘛！

​ 更爽的是，以后谁想得到用户登录的通知，只需：implements Observer —> 调用addObserver(Observer o)方法在我这里留下联系方式就行。

​ 我再也不用被动地修改代码。

​ 后来，帝国对Observable兄弟进行招安，也正式把这种多对一的订阅关系称为订阅/发布模型，也叫观察者模式。像DBContext这种观察某个事件发生的类被称为观察者，而像UserAction这种发布消息的类称为被观察者。
